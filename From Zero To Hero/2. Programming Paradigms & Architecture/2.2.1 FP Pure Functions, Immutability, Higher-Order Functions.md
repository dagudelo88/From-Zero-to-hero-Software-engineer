## 1. Introduction

### Why these concepts matter in real software engineering
Pure functions, immutability, and higher-order functions are the **three pillars** that make modern codebases reliable, scalable, and joyfully maintainable.

- **Predictability & Debugging**: A pure function always does the same thing for the same input. No more “it worked on my machine” bugs.  
- **Testability**: You can test a pure function in isolation with zero mocking. Companies like Netflix, Meta, and Stripe rely on this for their massive test suites.  
- **Concurrency & Performance**: Immutable data + pure functions = safe parallel execution (React, Redux, Elixir, Rust). No race conditions.  
- **Code Reuse & Composition**: Higher-order functions let you build powerful abstractions (map, filter, Redux reducers, React hooks). Interviewers love asking “Refactor this to use map/filter/reduce” because it separates senior engineers from juniors.

These aren’t academic toys — they power React, Redux, RxJS, Python’s pandas internals, and serverless functions at AWS.

### How they build on each other (learning roadmap)
1. **Pure Functions** → The foundation. Learn to eliminate side effects first.  
2. **Immutability** → Protects the data that pure functions operate on.  
3. **Higher-Order Functions** → The superpower that lets you compose pure, immutable operations into elegant pipelines.

Master them in this order and everything clicks.

## 2. Core Concepts

### Pure Functions

#### Theory (from zero to expert)
A function is **pure** if:
1. It always returns the same output for the same input (deterministic).  
2. It has **no side effects** — it doesn’t modify anything outside its scope (no global variables, no DOM changes, no file I/O, no `console.log` in production code, no mutating parameters).

Pure functions are mathematical functions in disguise.

**Real-world analogy**:  
Think of a coffee vending machine. You put in the same coins → you always get the same coffee. It never reaches into your wallet or changes the price based on the weather. That’s purity.

**Impure examples** (what you see every day):
```python
# Impure - modifies external state
counter = 0
def increment():
    global counter
    counter += 1
    return counter
```

#### Code Examples

**Python**
```python
# Pure
def add(a: int, b: int) -> int:
    return a + b

# Impure (side effect + non-deterministic)
import random
def impure_add(a, b):
    print("Adding...")          # side effect
    return a + b + random.randint(0, 10)
```

**JavaScript / TypeScript**
```ts
// Pure
const multiply = (x: number, y: number): number => x * y;

// Impure
let total = 0;
const impureMultiply = (x: number, y: number): number => {
  total += x * y;               // mutates external state
  return x * y;
};
```

#### Common pitfalls & how to avoid them
- **Accidental mutation of parameters** → Always work on copies (`[...arr]` or `Object.assign` or `structuredClone`).  
- **Hidden side effects** (logging, API calls) → Move them outside the function (inject dependencies).  
- **Date.now() or Math.random()** → Pass them as parameters instead.

#### Time & Space Complexity
Purity itself doesn’t change Big-O, but pure functions are easier to memoize (caching), turning O(2^n) recursive algorithms into O(n) with one line.

#### Practice Exercises

**Easy**  
Problem: Write a pure function `circleArea(radius)` that returns πr².  
Hint: Use `Math.PI`. No global variables.  
**Solution**:
```python
import math
def circle_area(radius: float) -> float:
    return math.pi * radius ** 2
```
Test cases: `circle_area(5) == 78.53981633974483`, `circle_area(0) == 0`.

**Medium**  
Problem: Refactor this impure function to pure (it modifies the input list).  
```python
def add_bonus impure(salaries):
    for i in range(len(salaries)):
        salaries[i] += 1000
    return salaries
```
**Solution** (Python):
```python
def add_bonus(salaries: list[int]) -> list[int]:
    return [salary + 1000 for salary in salaries]  # new list
```
Explanation: List comprehension creates a copy. Original list untouched.

**Hard**  
Problem: Implement a pure `memoize` decorator that caches results of any pure function.  
**Solution** (Python):
```python
def memoize(func):
    cache = {}
    def wrapper(*args):
        if args not in cache:
            cache[args] = func(*args)
        return cache[args]
    return wrapper

@memoize
def fibonacci(n: int) -> int:
    if n <= 1: return n
    return fibonacci(n-1) + fibonacci(n-2)
```
Explanation: Same input → instant lookup. Turns exponential recursion into linear time.

### Immutability

#### Theory (from zero to expert)
Immutability means **once data is created, it cannot be changed**. Any “modification” returns a brand-new copy.

**Real-world analogy**:  
A printed book vs a Google Doc. You can’t cross out words in a printed book — you have to publish a new edition. That’s immutable data.

Languages handle it differently:
- Python: tuples, frozensets, `frozendict` (3rd party)  
- JavaScript: `const` (shallow), `Object.freeze()`, libraries like Immer or Immutable.js

#### Code Examples

**Python**
```python
# Mutable (dangerous in FP)
lst = [1, 2, 3]
lst.append(4)          # changes original

# Immutable
tup = (1, 2, 3)
# tup[0] = 99  # TypeError - protected!
new_tup = tup + (4,)   # new tuple
```

**JavaScript / TypeScript**
```ts
// Shallow freeze
const obj = Object.freeze({ name: "Alice", scores: [10, 20] });
obj.name = "Bob";           // silently fails in strict mode
// obj.scores.push(30);     // still mutable! (shallow)

const deepImmutable = (obj: any) => 
  Object.freeze(JSON.parse(JSON.stringify(obj)));
```

#### Common pitfalls & how to avoid
- **Shallow vs Deep freeze** → Use Immer (auto-produces new state) or `structuredClone`.  
- **Performance myth** → Modern engines (V8, PyPy) optimize immutable patterns; don’t fear copies.  
- **Accidentally mutating in Redux/React** → Always use spread operator or Immer.

#### Time & Space Complexity
Each “mutation” costs O(n) space for the copy, but you gain thread-safety and easier caching.

#### Practice Exercises

**Easy**  
Problem: Write an immutable `addToCart(cart, item)` that returns a new cart.  
**Solution** (JS):
```js
const addToCart = (cart, item) => [...cart, item];
```
Test: Original `cart` unchanged.

**Medium**  
Problem: Update a user’s address without mutating the original object (deep).  
**Solution** (Python with dict):
```python
def update_address(user: dict, new_address: str) -> dict:
    return {**user, "address": new_address}
```
Or in JS: `const newUser = { ...user, address: newAddress };`

**Hard**  
Problem: Implement `immer`-style `produce` function that lets you write mutable-looking code but returns immutable result.  
**Solution** (simplified JS):
```js
function produce(base, recipe) {
  const draft = JSON.parse(JSON.stringify(base));
  recipe(draft);
  return Object.freeze(draft);
}

const nextState = produce({ count: 0 }, draft => { draft.count += 1; });
```
Explanation: You mutate the draft safely; original stays frozen.

### Higher-Order Functions (HOFs)

#### Theory (from zero to expert)
A higher-order function is any function that:
1. Takes one or more functions as arguments, OR  
2. Returns a function as its result.

They turn functions into first-class citizens (data you can pass around).

**Real-world analogy**:  
A remote control that can accept any “button press” function. The remote doesn’t care what the button does — it just executes it. That’s `map`, `filter`, `reduce`.

#### Code Examples

**Python**
```python
numbers = [1, 2, 3, 4]

# HOFs built-in
squares = list(map(lambda x: x**2, numbers))
evens   = list(filter(lambda x: x % 2 == 0, numbers))
total   = reduce(lambda x, y: x + y, numbers, 0)  # from functools
```

**JavaScript / TypeScript**
```ts
const numbers = [1, 2, 3, 4];

const squares = numbers.map(x => x ** 2);
const evens   = numbers.filter(x => x % 2 === 0);
const total   = numbers.reduce((acc, x) => acc + x, 0);
```

**Advanced: Function that returns a function (closure)**
```python
def make_multiplier(n):
    def multiplier(x):
        return x * n
    return multiplier

double = make_multiplier(2)
print(double(5))  # 10
```

#### Common pitfalls & how to avoid
- **Callback hell** → Chain with `.then()` or use async/await.  
- **Performance with large arrays** → Use generators (`yield`) or transducers for huge data.  
- **Losing `this` context** → Use arrow functions or `.bind()`.

#### Time & Space Complexity
- `map` / `filter` → O(n) time, O(n) space  
- `reduce` → O(n) time, O(1) extra space (if not building new structure)

#### Practice Exercises

**Easy**  
Problem: Use `map` to double every number in an array.  
**Solution** (Python): `list(map(lambda x: x*2, nums))`

**Medium**  
Problem: Implement `compose(f, g)` so `compose(square, add1)(5)` → `square(add1(5))` = 36.  
**Solution**:
```python
def compose(f, g):
    return lambda x: f(g(x))

def add1(x): return x + 1
def square(x): return x ** 2

result = compose(square, add1)(5)  # 36
```

**Hard**  
Problem: Build a custom `pipeline` HOF that accepts any number of functions and runs them left-to-right.  
**Solution** (JS):
```js
const pipeline = (...fns) => x => fns.reduce((acc, fn) => fn(acc), x);

const result = pipeline(
  x => x + 10,
  x => x * 2,
  x => x - 5
)(5); // → 25
```
Explanation: Reduces an array of functions into a single composed function — the foundation of Redux middleware and React hooks.

## 3. Summary & Mastery Section

### Key Takeaways
- **Pure Functions**: Same input → same output, zero side effects → predictable code.  
- **Immutability**: Never mutate; always return new data → safe concurrency and debugging.  
- **Higher-Order Functions**: Treat functions as data → build powerful, reusable pipelines.

### Comparison Table

| Concept            | Guarantees                  | Typical Use Cases                     | Performance Trade-off       | Interview Red Flag          |
|--------------------|-----------------------------|---------------------------------------|-----------------------------|-----------------------------|
| Pure Functions     | Deterministic + no side effects | Redux reducers, math libs, tests     | Easy memoization            | Mutates parameter or uses global |
| Immutability       | Data never changes          | React state, Redux, concurrent code  | Extra memory (mitigated by structural sharing) | Direct `obj.prop = x` after freeze |
| Higher-Order Functions | Abstraction & composition | map/filter/reduce, hooks, middleware | O(n) per operation          | Nested for-loops instead of HOFs |
